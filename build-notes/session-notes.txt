Decisions Made That Aren't in the Master Prompt

Cache namespace sd5_ — bumped from your existing sd4_. The new architecture's parseCSV output shape is identical but the module structure is different enough that stale cache could cause subtle bugs. Bumping the namespace means every user gets a clean cache on first load of the new site with zero code needed.
loadCSS(href) in js/utils.js — The master prompt says section CSS is loaded lazily by each view module but doesn't specify the mechanism. Decision: each view calls loadCSS('/css/blog.css') at the top of its render function. loadCSS is idempotent (tracks injected hrefs in a Set) and returns a Promise so views can await loadCSS(...) before rendering if needed. In practice, CSS loads fast enough that views don't await it.
chordsPerPage: 12 — The master prompt specifies 12 per page for chords but doesn't say where this config lives. It's in CFG in js/api.js, matching the pattern of postsPerPage: 6.
renderTabSSR in worker/utils.js — The master prompt specifies SSR for chord pages but doesn't detail the tab rendering algorithm for the server side. Decision: chords SSR renders tab content as a <pre> block with chord names on a line above lyrics (plain text, no [G] tokens), which is maximally crawler-friendly and readable without JS.
prefetchSheets in js/api.js — Not in the master prompt. Added as a fire-and-forget cache warmer called on home page load. Warms blog, skills, projects so navigating to those sections feels instant. Does nothing if cache is already fresh.
buildHomeSchemas in js/seo.js — The master prompt specifies Person + Organization + WebSite schema on home but doesn't say whether they should be static in index.html or dynamic. Decision: they're in js/seo.js as buildHomeSchemas() called by updateSEO on the home route. This means index.html can be a clean shell without inline schemas (the static schemas in your current index.html will be removed when we build index.html).
sanitizeHTML uses regex in worker/utils.js, DOMParser in js/utils.js — Different implementations for different environments. Workers runtime has no DOMParser. Both implementations have the same safe tag/attribute whitelist. Next Claude needs to keep these in sync if the whitelist changes.
removeSchemas cleans ['dyn-schema', 'chord-schema', 'faq-schema'] on every updateSEO call — Proactive cleanup ensures no schema leakage between routes (e.g. navigating from a chord page to a blog page never leaves a MusicComposition schema behind).
